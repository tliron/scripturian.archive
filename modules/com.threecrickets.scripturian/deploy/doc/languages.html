Supported Languages

Python

Two implementations exist for Python on the JVM: Jython and Jepp. The former is 100% Java, while the latter uses a Java bridge to a natively compiled CPython of your choice, and is generally more complicated to install and performs poorly.

Jython 2.5.2 is supported via its JSR-223 interface, with some fixes for odd behavior. Compilation to byetcode is supported, though it is currently cached to memory and thus executables are re-compiled every time the JVM is restarted.

Though 2.5.2 is not yet available at the time of this writing, you can get a recent snapshot. Unfortunately, Jython 2.5.1 could not support Scripturian's threading model over JSR-223. We are grateful to the Jython team for accepting our suggestion on how to fix it!

Jepp is supported via its JSR-223 interface. It does not support invocation of entry points.

Ruby

Ruby is implemented by JRuby over our custom interface. Version 1.4 is supported. Compilation to byetcode is supported, though it is currently cached to memory and thus executables are re-compiled every time the JVM is restarted.

JRuby comes with JSR-223 support, and configurable threading models. However, after considerable experimentation, we found all of them incompatible with Scripturian's requirements, and so devised our own adapter. Ours is based on creating separate JRuby runtimes per execution context, with a trick to allow changing the standard output and standard error and thus reusing the runtime.

You will need a snapshot of JRuby 1.5 if you wish to combine JRuby and Jython on the same VM. We are grateful to the JRuby team for accommodating our request to allow this, and cooperating with the Jython team to make it happen!

Groovy

Groovy is designed for the JVM and works quite transparently via its built-in JSR-223 interface.

Clojure

Clojure is designed for the JVM and implemented via our custom interface. Our implementation involves creating lightweight namespaces on-the-fly for each execution context. Clojure compilation to JVM bytecode is not supported, but it remains an open question, too, whether this would significantly improve performance. Clojure's highly dynamic nature would make compiled code quite uninteresting. It's likely that the JVM's optimization would work just as well on "interpreted" Clojure.

A JSR-223 interface exists, but we found it unusable for Scripturian's concurrency requirements.

PHP

PHP5 is implemented by Caucho's Quercus. Note that Quercus license is GPL, which puts limitations on your ability to distribute it with your product.

Scripturian support is over Quercus' included JSR-223 interface. It does not support invocation of entry points.

Note that Scripturian replaces PHP's scriptlet scheme with its own. Usage and performance should be identical, but Scripturian's scheme is more powerful than PHP's, because it allows for mixing languages in one file.

JavaScript

Your distribution of Java may already have JavaScript support built in, in which case you're already good to go. This is true for Sun's JDK and OpenJDK. Otherwise, JavaScript is implemented by Mozilla's Rhino.

Note that Sun and OpenJDK's built-in JavaScript support is a simplified version of Rhino, which may limit the power of Scripturian, so that Mozilla's version is recommended. If you have both versions installed, use the "rhino-nonjdk" name to circumvent the built-in Rhino (see demo JavaScript source code).

Mozilla's Rhino requires a small library for Java scripting standard (JSR-223), which is available at the Java scripting project. (This was true for version 1.7R1.) See the requirement notes if you are using Java 5.

Succinct

Succinct is currently the only language with built-in Scripturian support!

Velocity

Get Velocity here. Note that you also need a small library for Java scripting standard (JSR-223), which is available at the Java scripting project.
